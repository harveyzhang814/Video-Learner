# ADR: 字幕翻译分块与顺序上下文方案

**日期**: 2026-06-21
**状态**: accepted

## 背景

为英文字幕（`original_en.md`）新增 LLM 翻译步骤（`translate` 节点）。`original_en.md` 的结构是每 2-3 秒一个 `## HH:MM:SS` 时间戳块，每块只是句子的一个碎片，不能直接逐块送入 LLM——语义不完整、翻译质量极低。

同时，ASR 输出可能完全没有标点（逗号、句号均无），导致无法用标点检测句子边界。

参考来源：Vimeo Engineering Blog《How We Built AI-Powered Subtitles at Vimeo》（Tanushree Nori, 2026-01），该文描述了一套针对"LLM 自由翻译破坏时间戳同步"问题的三阶段分离架构。

## 决策

采用改编自 Vimeo 的两阶段方案，**不实现 Vimeo 的 Phase 3（行数映射）**：

### Phase 1：时间窗口分块（改编自 Vimeo Phase 1）

与 Vimeo 不同，本项目的分块**不依赖标点检测**（ASR 输出无标点），改用时间窗口：

- 将相邻 `## HH:MM:SS` 碎片按 **20-30 秒时间窗口**合并为一个"思想块"
- 若块内恰好存在标点，优先在句末切割（bonus 信号）
- 单块字符数硬上限 800，防止 LLM 上下文过载
- 每块保留 `start_ts`（第一个碎片的时间戳）

### Phase 2：顺序翻译 + 前块末尾接续（本项目独有）

Vimeo Phase 2 是纯"创意翻译"（不约束行数），本项目采用相同思路，但加入**顺序前置上下文**以解决块间断句问题：

```
--- 已翻译上文（末尾，接续参考）---
{zh_prev_tail}     ← 上一块中文输出的最后 100-200 字符

--- 待翻译 ---
{merged_en_text}

--- 下文参考（只读，不翻译）---
{next_chunk_en}    ← 下一块英文，帮助 LLM 预判语义走向
```

`zh_prev_tail` 成本恒定（不随块数增长），LLM 从前块语义节点自然接续，无需判断断句位置。

### 不实现 Vimeo Phase 3（行数映射）

Vimeo Phase 3 的目的是把自由翻译的文本"重新塞回"与原文对齐的时间槽，解决时间戳同步问题。

**本项目明确不需要时间戳对齐**：`original_zh.md` 的时间戳粒度（每 20-30 秒一个）可以与原文（每 2-3 秒一个）完全不同，这是预期行为。省略 Phase 3 同时省去了 Vimeo 95% 首次通过率的限制和回退链路的工程复杂度。

## 理由

| 方案 | 排除原因 |
|------|----------|
| 逐块翻译（无合并）| 每块仅 2-3 秒，语义不完整，翻译质量不可接受 |
| 标点边界切割 | ASR 输出无标点，此方法在本场景无效 |
| 传入全部已翻译上文 | LLM 上下文随块数线性增长，长视频会超出 token 限制 |
| Vimeo Phase 3（行数映射）| 不需要：用户明确说时间戳无需与原文对齐 |
| 完整 Vimeo 三阶段 | Phase 3 的工程复杂度（重试、降级、确定性回退）对本场景过度 |

## 影响

- `translate` 步骤串行调用 LLM（顺序执行是硬性约束，zh_prev_tail 依赖前序输出）
- 超时设置 60 分钟（随 `timeout_scale` 缩放），适用 1-3 小时视频需使用 `--long`
- 输出 `original_zh.md` 时间戳远少于原文（每句一个 vs 每 2-3 秒一个），`md2vtt` 现有逻辑可正常处理
- 实现位置：`scripts/translate_subs.sh`（新文件）
- 参考规格：`docs/superpowers/specs/2026-06-21-subtitle-translate-design.md`
