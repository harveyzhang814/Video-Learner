## 错误类型总结

- **意图含糊/上下文缺失**：只描述“界面某处有问题”，但缺少页面/组件路径、期望值、当前值、复现步骤（如是否拖拽过 resizer）、以及“上限/下限”是视觉感受还是明确的像素限制，导致排查范围过大。

## 典型示例

### 示例 1：UI 宽度被限制但未给上下文

- **原 query**：GUI的播放界面（右边栏）的宽度有上限，太小了，查看容器和子容器，是什么制约了宽度

- **Refined Query（推荐可复制）**：

```text
请帮我定位 Electron GUI 里右侧播放栏（video panel）的宽度为什么有“最大值”限制：
- 入口文件/页面：electron/src/renderer/index.html
- 现象：拖动中间的竖向 resizer 时，右侧栏宽度最多只能到 <xxx> px（或“明显到不了更宽”）
- 期望：最大宽度至少允许到 <yyy> px，或跟随窗口宽度自适应
请指出具体是哪条 CSS 或 JS clamp/计算逻辑在限制（给出代码行号范围），以及如何调整最大值/改为可配置。
```

- **原 query 的缺陷**：
  - **意图含糊/上下文缺失**：没有给出具体入口文件、是否通过拖拽触发、以及“上限”是主观感受还是像素级 hard limit。

- **推荐 query 好在哪里**：
  - **可直接定位到规则来源**：明确入口文件与交互（resizer），能快速找到 `--video-panel-width` 与 `clampVideoPanelWidth()` 之类的限制点。
  - **把“问题”转成“可验证的约束”**：用当前值/期望值描述，方便给出精确修改方案（改 max、改成配置、或改成百分比/自适应）。
